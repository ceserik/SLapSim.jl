\chapter{Conclusion and future work}
\section{Conclusion}
\setlength{\parindent}{2em}
\raggedbottom
The goal of investigation of available numerical methods for optimal control was accomplished. The topic proved to be remarkably deep with all of the subproblems: dynamic system modeling, transcription, algebraic modeling, mesh refinement, NLP solving, sparse linear solving, sensitivity analysis and frameworks that encompass them being substantial fields in their own right. The main challenge was to understand these topics in a meaningful way without spending a disproportionate amount of time on a single topic.

As for implementation, Julia was an interesting language to learn. It offers fast execution, but that comes with caveats. Debugging is considerably more complicated. On a smaller project the debugging is manageable, but on a larger project it is not user friendly. Debugging in Python and MATLAB is much easier. Julia's main selling point is that it is a just-in-time compiled language, rivaling C after compilation. This is true, but only if the code is written with performance in mind. Otherwise it can be as slow as Python.

At the beginning of this project I wanted a more complex model, going further in complexity than my bachelor thesis. This would include adding pitch and roll, modeling the suspension with a spring and damper for each wheel, adding a 1RC equivalent circuit model for the accumulator, properly implementing energy accumulation, and accounting for road bank and inclination. Then try to find the limits of model complexity that can be simulated. Unfortunately, there was not enough time for this. Also due to lack of time I did not manage to investigate collocation methods based on Chebyshev nodes.

Trusting this tool completely, not using common sense and designing a car according to this tool would most likely lead to an undriveable vehicle, as the objective of time minimization does not account for stability or driveability of the vehicle. The work in~\cite{perantoniOptimalControlFormula2014} also addresses this issue. The optimal driving produced by the NLP is unachievable for a real driver, since the vehicle operates at its limit, where a single mistake would overload the tires and cause failure. Also the validation of this tool was only done using ODE solvers through mesh refinement, a proper validation using the data from a real vehicle on a race track was not done due to time constraints.

In the thesis I have found that the transcription for fastest solving was Legendre-Gauss-Lobatto in integral formulation, but the accuracy needs to be investigated further as the two other methods had more similar results. The fastest NLP solver was MadNLP on CPU, and the fastest linear solver was ma97, with the one failed problem due to iteration limit. Automatic differentiation is superior to the quasi-Newton derivative estimation. The resulting tool can optimize a lap for the tested vehicle models in seconds to minutes, making it usable for concept evaluation, perform sensitivity analysis and find optimal value of a parameter. The code is open source on GitHub \url{https://github.com/ceserik/SLapSim.jl}, it supports multiple collocation schemes and solver backends with adaptive mesh refinement, and it is built on a modular vehicle framework.

\section{Future Work}
For NLP solvers a better set of options can be found for Ipopt and MadNLP by investigating why the solver fails in some cases. It may also make sense to try Uno solver~\cite{vanaret_implementing_2025} and construct a new specialized solver for this problem, but that would take a substantial amount of time.

Building a model for optimization was difficult because debugging is hard. When the grid is sampled too densely or the problem is otherwise poorly conditioned, Ipopt fails to converge. I was not using any tool to find which constraint is causing this. Debugging could be improved by writing or using a tool which would tell which constraint is causing issues. This would significantly decrease the development time.

Automated benchmarking is essential, because small changes can break many things, and a bug may go unnoticed until the simulation crashes with a different track and a different vehicle long after the bug was introduced. In future this should be added to GitHub CI/CD.

The implemented mesh refinement method is the simplest version of mesh refinement, comparison with an ODE solver is not ideal. GPOPS-II uses more advanced methods. I believe the largest gains in solve time and solver success rate can be made by a better mesh refinement method, currently meshes can only be split, not merged, which has caused numerical difficulties after a couple of iterations of refinement.

It is worth investigating the possible implementations of the losses from Section~\ref{motormodel} and the use of softmax or other approximations. The current simulation vehicle model does not account for gearboxes with discrete stages or for combustion engines either, both of which could be incorporated within the existing modeling framework and which I would like to add.

JuMP spent 90\% of the NLP solving time computing derivatives when the constraints had dense expressions~\cite{julia_discourse_ipopt_hessian}, leaving only 10\% for the actual solver. CasADi does not appear to have this issue. Implementing the automatic differentiation with ExaModels~\cite{shin2024accelerating} on the GPU may solve this.

One thing that was not a goal, but would have been very useful, is a validation of the sensitivity analysis against real-world data. This would involve driving the vehicle on a track and measuring the lap time, adding extra weight to the car (sandbags, or myself as a passenger), driving again, and comparing the times. I had wanted to do this for a long time, but there was not enough time available.

The comparison method using the mean and median is a misleading representation for benchmarking as it is skewed by a few problems with a long solving time. A standard way to compare solvers is in \cite{dolanBenchmarkingOptimizationSoftware2004}. This should be done in the future. Also the errors between different transcription methods should be examined.
